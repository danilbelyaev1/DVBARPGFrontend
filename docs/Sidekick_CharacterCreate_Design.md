# Sidekick: настройка внешности в CharacterCreate и отображение в Run

## Что установлено в Assets/Synty/

- **SidekickCharacters** — ядро: Runtime API, SQLite-база видов/частей/пресетов, демо.
- **Структура:**
  - `Scripts/Runtime/API/SidekickRuntime.cs` — создание персонажа из списка частей (SkinnedMeshRenderer), блендшейпы (пол/вес/мускулы), цвета.
  - `Scripts/Runtime/Database/` — DTO: SidekickSpecies, SidekickPart, SidekickPartPreset, SidekickBodyShapePreset, SidekickColorPreset и т.д.
  - `Scripts/Runtime/Serialization/` — **SerializedCharacter** (Name, Species, Parts, ColorSet, ColorRows, BlendShapes), SerializedPart, SerializedBlendShapeValues (BodyTypeValue, BodySizeValue, MuscleValue), SerializedColorSet/SerializedColorRow.
  - `Resources/Meshes/SK_BaseModel` — донор рига. Части ищутся по маске файлов `SK_*_*_*_*_*.fbx` в Assets.
- **Виды (Species)** — в БД (sk_species); части привязаны к виду (BASE-части по видам). Сейчас у тебя эльфы — скорее всего один вид "Elf" или аналог.
- **Типы частей (CharacterPartType):** Head, Hair, Eyebrow*, Eye*, Ear*, FacialHair, Torso, Arms, Hands, Hips, Legs, Feet, Attachment* и т.д.
- **Группы для пресетов (PartGroup):** Head, UpperBody, LowerBody.
- **Демо:** RuntimePresetDemo (пресеты головы/верх/низ + BodyShape + Color), RuntimePartsDemo (выбор по частям + слайдер веса). Оба загружают модель из Resources, собирают список мешей и вызывают `SidekickRuntime.CreateCharacter(...)`.

Идея под классы: для каждого класса свой набор ассетов (свой вид или фильтр частей). Сейчас только эльфы (лучник); воин и маг — отдельные паки, позже. Нужна привязка класс → вид/набор (например vanguard → species_id воина, hunter → elf, mystic → вид мага).

## Задумка (кратко)

- Настройка **только внешности** (без одежды): пол, вес/телосложение, лицо, волосы и всё остальное по каталогу Sidekick.
- Сцена **CharacterCreate**: полный UI настройки + превью персонажа через Sidekick; при «Создать» — сохранение выбора на бэке и создание персонажа.
- Сцена **Run**: играем за этого же персонажа — подставляем сохранённую внешность (тот же набор частей/пресетов/блендшейпов).
- Для каждого **класса** свой набор (вид/пак). Сейчас только один пак (эльфы) — закладываем маппинг класс → species/набор, чтобы позже подставить воина/мага.

---

## Предлагаемая модель данных (внешность, без одежды)

Использовать формат, близкий к **SerializedCharacter** Sidekick, чтобы можно было сохранять/загружать и применять через их API:

- **SpeciesId** (int) — id вида в Sidekick (класс у нас мапим на вид: hunter → elf и т.д.).
- **Gender** — уже есть; маппим в блендшейп BodyType (masculineFeminine): например male → 0, female → 100 (или наоборот, по документации).
- **BlendShapes:** BodyTypeValue (пол), BodySizeValue (вес: минус = skinny, плюс = heavy), MuscleValue (мускулатура). Лицо — либо отдельные блендшейпы (если Sidekick отдаёт), либо пресет головы.
- **Parts** — список `{ PartType, PartName }` по всем CharacterPartType (голова, волосы, торс, руки, ноги и т.д.). Без одежды = только BASE-части или «голые» пресеты.
- **Colors** — ColorSet + ColorRows (или id цветового пресета), чтобы сохранять кожу/волосы и т.д.
- **Пресеты вместо частей (опционально):** можно хранить HeadPresetId, UpperBodyPresetId, LowerBodyPresetId вместо списка частей — тогда при загрузке берём пресеты из БД Sidekick и собираем части из них (как в RuntimePresetDemo).

Для бэка достаточно одного JSON-поля (например `appearance_json`) в таблице персонажей или отдельной таблицы `character_appearance` с полем того же формата. Контракт: при создании/обновлении персонажа клиент присылает appearance; при запросе персонажа/профиля бэк отдаёт appearance обратно.

## Цепочка на клиенте

1. **CharacterCreate**
  - Выбор класса → определяем Species (или набор частей) для этого класса.
  - Выбор пола → задаём диапазон/значение для BodyType (блендшейп пола).
  - Кастомизация: вес (BodySize), мускулатура (Muscle), лицо (пресет головы или части), волосы (часть Hair), остальное по желанию (брови, уши и т.д.), цвета (пресет или ColorRows).
  - Превью: один раз при открытии экрана и при каждом изменении вызываем SidekickRuntime (загрузка базы/частей по Species, сбор списка мешей из выбранных частей/пресетов, установка BlendShapes и цветов, CreateCharacter). Модель показываем в сцене (камера на превью).
  - По кнопке «Создать»: сериализуем текущий выбор в SerializedCharacter (или наш DTO того же формата), отправляем на бэк вместе с name, classId, gender (CreateCharacter + сохранение appearance).
2. **Run**
  - При старте забега для локального игрока: запрашиваем с бэка профиль/персонажа с appearance (или уже есть в IProfileService после выбора персонажа).
  - По appearance восстанавливаем SidekickRuntime (Species, Parts, BlendShapes, Colors) и вызываем CreateCharacter.
  - Полученный GameObject подставляем как визуал игрока: либо заменяем дочерний меш у текущего корня игрока (NetworkPlayerReplicator), либо делаем корень игрока контейнером и создаём Sidekick-персонажа как дочерний объект с нужным ригом/аниматором.
3. **Класс → набор**
  - Сейчас: один пак (эльфы) → один Species (или один набор BASE-частей). Класс hunter мапим на этот вид.
  - Позже: vanguard → пак воина (другой Species/папка), mystic → пак мага. В коде — маппинг classId → speciesId (или путь к Resources/конфиг с перечнем частей).

## Решения (зафиксированы)

- **Бэк:** с нуля — миграция `appearance` (JSON, nullable) в таблице `characters`; при создании персонажа принимается `appearance` (опционально); в ответах list/character возвращается `appearance`.
- **Пол:** хранится на бэке (gender); на клиенте мапим в BodyTypeValue при сборке (male/female → блендшейп).
- **Лицо:** точная настройка — в DTO есть `faceBlendShapes` (имя блендшейпа → вес); после CreateCharacter выставляем веса по имени.
- **Волосы/борода:** в общем списке `parts` (PartType + PartName) — один формат для всех частей, оптимально по размеру и единообразию.
- **Цвета:** храним `colorPresetId` (int) — компактно; при сборке подставляем пресет из БД Sidekick.
- **Превью:** один ракурс с возможностью крутить персонажа по ПКМ (отдельный скрипт на камеру/пивот).
- **Run:** собираем персонажа из Sidekick по сохранённому appearance и подставляем его как визуал игрока (тот, кого создали).
- **Редактирование внешности:** пока только при создании.

### Блок A: Сцена CharacterCreate — превью и вращение

1. **Открыть сцену CharacterCreate**
  Файл: `Assets/Scenes/CharacterCreate.unity`.
2. **Площадка для превью**
  - Создать пустой GameObject, назвать например `PreviewRoot`.  
  - Под ним создать дочерний пустой объект `PreviewPivot` (сюда будет ставиться собранный Sidekick-персонаж).  
  - Позиция `PreviewRoot` — где тебе удобно (например центр экрана по X/Z, Y = 0).  
  - Масштаб оставить (1, 1, 1).
3. **Камера превью**
  - Либо использовать основную камеру сцены, направив её на `PreviewPivot`.  
  - Либо создать отдельную камеру только для превью (например дочернюю к `PreviewRoot` или к своей UI-камере).  
  - Важно: камера должна смотреть на точку, где стоит персонаж (на `PreviewPivot` или на сам персонаж).
4. **Вращение по ПКМ**
  - Выбрать объект, который должен крутиться при зажатой ПКМ:  
    - **Вариант А:** вешать скрипт на `PreviewPivot` — тогда крутится персонаж на месте.  
    - **Вариант Б:** вешать на камеру — тогда крутится камера вокруг сцены.
  - Добавить компонент **PreviewRotateOnDrag** (скрипт: `Assets/Game/CharacterCreation/PreviewRotateOnDrag.cs`).  
  - В инспекторе:  
    - `Sensitivity Y` — скорость вращения по горизонтали (например 0.3).  
    - `Rotate Vertical` — включить только если крутишь камеру и нужен наклон вверх/вниз.  
    - `Sensitivity X` — только при включённом Rotate Vertical.
5. **Связь сборщика с превью**
  - Найти или создать в сцене объект-«менеджер» для экрана создания (например тот же объект, где висит `CharacterCreateScreen`, или отдельный GameObject).  
  - Добавить на него компонент **SidekickAppearanceBuilder** (скрипт: `Assets/Game/CharacterCreation/SidekickAppearanceBuilder.cs`).  
  - В коде экрана создания нужно будет:  
    - хранить ссылку на `SidekickAppearanceBuilder` и на `PreviewPivot` (Transform или GameObject);  
    - при открытии экрана и при смене класса/пола/опций вызывать `builder.BuildAppearance(defaultOrCurrentAppearance, onDone)` и в `onDone` ставить полученный GameObject в позицию `PreviewPivot`, делая его дочерним к пивоту (старый превью при этом уничтожать).  
     Это будет сделано в коде в следующих шагах; здесь достаточно завести объект с `SidekickAppearanceBuilder` и запомнить, куда вешать превью (`PreviewPivot`).

### Блок B: CharacterCreate — UI и данные внешности

1. **UI для тела (вес / мускулатура)**
  - В панели кастомизации (или рядом с полом) добавить два слайдера:  
    - «Вес» (BodySize): минимум например -1, максимум 1, значение по умолчанию 0.  
    - «Мускулатура» (Muscle): те же диапазоны при желании или 0…1.
  - Подписать их в `CharacterCreateScreen`: поля типа `Slider` (или два слайдера + два TMP для подписей).  
  - При изменении значения — обновлять текущий DTO внешности и заново вызывать `BuildAppearance` для превью.
2. **UI для волос (и опционально борода)**
  - Выпадающий список (Dropdown) или набор кнопок: выбор части Hair из списка частей Sidekick для выбранного вида (Species).  
  - Список вариантов волос нужно получать из Sidekick (DatabaseManager → части с типом Hair для текущего Species).  
  - При выборе — добавлять/заменять запись в `CharacterAppearanceData.Parts` по типу Hair (и при необходимости FacialHair), затем пересобирать превью.
3. **UI для цветов**
  - Выпадающий список или кнопки: выбор цветового пресета по id (список пресетов из Sidekick DatabaseManager).  
  - При выборе — выставлять `CharacterAppearanceData.ColorPresetId`, пересобирать превью.
4. **Лицо (морфы)**
  - Если хочешь тонкую настройку: список имён блендшейпов лица из мешей Sidekick для головы + слайдеры (имя → 0…100).  
  - Значения писать в `CharacterAppearanceData.FaceBlendShapes` (Dictionary<string, float>).  
  - Можно начать с одного пресета головы (одна часть Head в Parts) и позже добавить слайдеры по морфам.
5. **Дефолтная внешность при открытии и смене класса/пола**
  - При открытии экрана CharacterCreate (и при смене класса или пола) собирать **дефолтный** `CharacterAppearanceData`:  
    - SpeciesId — из класса через `ClassSidekickSpeciesMap.GetSpeciesNameForClass(classId)` и поиск вида в Sidekick DB по имени.  
    - Parts — BASE-части для этого вида (из БД Sidekick по виду).  
    - BlendShapeValues: BodyType — из пола (male → 0, female → 100 или как в документации Sidekick), BodySize/Muscle — 0 или из слайдеров.  
    - FaceBlendShapes — пусто или дефолтные значения.  
    - ColorPresetId — первый пресет или 0, если так принято.  
    - Этим же дефолтом инициализировать текущее состояние UI (слайдеры, выбранные части, пресет цвета).  
    - Вызвать `BuildAppearance(defaultAppearance, onDone)` и показать превью на `PreviewPivot`.
6. **Кнопка «Создать»**
  - По нажатию: из текущего состояния UI собрать финальный `CharacterAppearanceData` (части, блендшейпы, морфы лица, ColorPresetId, SpeciesId).  
    - Сериализовать в JSON (например через JsonUtility или тот же формат, что ждёт бэк).  
    - Вызвать `meta.CreateCharacter(session, name, classId, _selectedGender, appearanceObject, result => { ... })`.  
    - В `CharacterCreateScreen` уже есть вызов `CreateCharacter` с `null` — заменить на передачу этого `appearanceObject`.

### Блок C: Run — подстановка персонажа по appearance

1. **Откуда брать appearance в Run**
  - После выбора персонажа в лобби/меню у тебя должен быть доступ к выбранному персонажу (например `SelectedCharacterId` и список персонажей из `IRuntimeMetaService` / Profile).  
    - В объекте персонажа из списка есть поле `Appearance` (из ответа API list characters или character graph).  
    - При старте забега (или при спавне локального игрока) взять этот объект и десериализовать в `CharacterAppearanceData` (если пришёл JObject — преобразовать в наш DTO).
2. **Где спавнится игрок**
  - Найти в сцене Run место, где создаётся/появляется объект игрока (префаб с `NetworkPlayerReplicator` или аналог).  
    - Корень игрока — один GameObject (с движением, сетью, аниматором).
3. **Подмена визуала**
  - В момент, когда локальный игрок создан (или сразу после получения appearance):  
    - Вызвать `SidekickAppearanceBuilder.BuildAppearance(characterAppearanceData, onDone)`.  
    - В колбеке `onDone(GameObject builtCharacter)`:  
      - Поставить `builtCharacter` как дочерний объект к корню игрока (или заменить существующий дочерний меш).  
      - Позиция/масштаб: локально (0,0,0) и (1,1,1), чтобы совпадать с ригом.  
      - Если у игрока уже есть старый визуал (меш/модель) — отключить или уничтожить его, чтобы остался только Sidekick-персонаж.
    - Аниматор: если у Sidekick-персонажа свой Animator с ригом, привязать его к тому же контроллеру, что использует движение игрока (или скопировать контроллер на Sidekick-модель).  
    - Итог: играем за того же персонажа, которого создали в CharacterCreate (тот же вид, пол, волосы, цвета).
4. **Если appearance пустой**
  - Если у персонажа нет сохранённого appearance (старый персонаж или null): использовать запасной вариант — дефолтную внешность по классу (как в п. 10) или показывать стандартный префаб без Sidekick, чтобы игра не падала.
